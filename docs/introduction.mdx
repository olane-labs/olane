Olane lets you build **tool nodes**: small services with declared methods that any agent — human or AI — can discover and call by name. Every node lives at an `o://` address. Every call is one primitive: `node.use(address, { method, params })`. The runtime handles peer discovery, routing, encryption, and lifecycle.

This page gives you the mental model in three minutes. When you're ready to build, jump to the [Quickstart](/getting-started/quickstart).

## The shape of an Olane program

```ts
import { oNodeTool } from '@olane/o-tool';
import { oNodeAddress } from '@olane/o-node';
import { oRequest } from '@olane/o-core';
import { CURRENCY_METHODS } from './methods/currency.methods.js';

class CurrencyConverter extends oNodeTool {
  constructor() {
    super({
      address: new oNodeAddress('o://currency'),
      leader: null,
      parent: null,
      description: 'FX conversion utility',
      methods: CURRENCY_METHODS,
    });
  }

  async _tool_convert(request: oRequest) {
    const { amount, from, to } = request.params;
    const rate = await this.fetchRate(from, to);
    return { converted: amount * rate, rate };
  }
}

const node = new CurrencyConverter();
await node.start();
```

Any other node — in the same process, the same network, or across a federation boundary — can now call it:

```ts
const r = await caller.use(new oAddress('o://currency'), {
  method: 'convert',
  params: { amount: 100, from: 'USD', to: 'EUR' },
});

console.log(r.result.data); // { converted: 92, rate: 0.92 }
```

## The three layers

<Frame>
```
┌──────────────────────────────────────────────────────┐
│  Agents          humans (CLI/web)  +  AI (LLMs)      │  ← users
├──────────────────────────────────────────────────────┤
│  Tool nodes      oNodeTool · oLaneTool · McpTool     │  ← your code
├──────────────────────────────────────────────────────┤
│  OS runtime      o-core · o-node · o-lane · o-os     │  ← infrastructure
└──────────────────────────────────────────────────────┘
```
</Frame>

You write the middle layer. Agents — human or AI — talk to your nodes through the same interface, in natural language or programmatic calls. The runtime handles everything below your code: libp2p networking, address resolution, the leader registry, connection pooling.

## The five primitives

<AccordionGroup>
  <Accordion title="Address — the name of a thing">
    `oAddress` (and the dialable `oNodeAddress`) is the only way one node refers to another. Addresses live in a namespace shaped like `o://parent/child`. They are names, not locations — the runtime resolves them to peers at call time.
  </Accordion>
  <Accordion title="Node — an addressable participant">
    A Node has an address, a libp2p PeerId, transport multiaddrs, and a lifecycle. Every participant in the network is a Node, including the leader.
  </Accordion>
  <Accordion title="Tool — a Node with declared methods">
    A Tool exposes capabilities through methods named with the `_tool_` prefix. Method metadata makes them discoverable to AI agents without code inspection.
  </Accordion>
  <Accordion title="Request and Response — the only message shape">
    A Request names a method and carries params. A Response carries `result.success`, `result.data`, and `result.error`. Every call yields exactly one response (or one stream that terminates). No fire-and-forget.
  </Accordion>
  <Accordion title="Method — a declared capability">
    `oMethod` describes a method's name, description, parameters (with types and required flags), and dependencies. The base class validates params against this declaration before your handler runs.
  </Accordion>
</AccordionGroup>

## The three iron rules

<CardGroup cols={3}>
  <Card title="Use node.use()" icon="phone">
    Every interaction with an address goes through `node.use()`. There is no other primitive.
  </Card>
  <Card title="Throw, don't return errors" icon="triangle-exclamation">
    Throw exceptions on failure. Return raw data on success. The base class builds the response envelope.
  </Card>
  <Card title="Read response.result.*" icon="magnifying-glass">
    Three fields nested under `result`: `success`, `data`, `error`. The most common mistake is reading them at the top level.
  </Card>
</CardGroup>

## When to use Olane

<CardGroup cols={2}>
  <Card title="You need agent-discoverable services" icon="users">
    Tool nodes are addressable by name and self-describing — AI agents can discover and call them without prior integration.
  </Card>
  <Card title="You need cross-process / cross-host calls" icon="network-wired">
    libp2p underneath gives you P2P networking with TCP, WebSocket, WebTransport, and circuit relay — no separate API gateway needed.
  </Card>
  <Card title="You're building multi-node coordination" icon="diagram-project">
    Parent-child hierarchies give you state isolation, lifecycle cascade, and namespace scoping for free.
  </Card>
  <Card title="You want one interface for humans and AI" icon="message-lines">
    The same tool node serves a CLI user, an HTTP client, and an LLM through one natural-language-or-method-call surface.
  </Card>
</CardGroup>
