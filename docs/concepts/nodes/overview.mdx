title: "What is a node?"
description: "Every participant in an Olane network is a node. Here's what that means."
---

A node is a thing with an address, a peerId, and a lifecycle. Your custom tool is a node. The leader is a node. A browser client is a node. The shape changes — what listens, what dials — but the contract is the same.

## The four things every node has

Every node in Olane carries exactly four pieces of state.

<ParamField path="address" type="oAddress" required>
  The `o://` name the node answers to. Constructed from `@olane/o-core`'s `oAddress`. Hierarchical paths like `o://company/finance/analyst` are first-class.
</ParamField>
<ParamField path="peerId" type="PeerId" required>
  The libp2p identity. Generated from a fresh keypair, or deterministically from a `seed` config so the same node has the same peerId across restarts.
</ParamField>
<ParamField path="multiaddrs" type="Multiaddr[]" required>
  Transport contact info — the actual `tcp`, `ws`, or `webtransport` endpoints other peers can dial. Populated when libp2p starts listening.
</ParamField>
<ParamField path="state" type="oNodeState" required>
  A state machine: `STOPPED → STARTING → RUNNING → STOPPING → STOPPED`. Read it with `node.state`.
</ParamField>

The `address` is human-meaningful and stable. The `peerId` is cryptographic and (optionally) stable. The `multiaddrs` describe where the peer can actually be reached right now. The leader registry binds the three together.

## How a node finds its peers

Olane does **not** use mDNS or a hardcoded bootstrap list. Discovery is hierarchical and registry-backed.

<Steps>
  <Step title="Bootstrap from the leader pointer">
    Every node is configured with a leader address. On start, it dials the leader directly using the leader's known multiaddrs.
  </Step>
  <Step title="Query the leader's registry">
    The leader hosts an `o://leader/registry` node. When another node wants to resolve `o://anything`, it queries the registry through `oSearchResolver`.
  </Step>
  <Step title="Fall through the resolver chain">
    If the registry has no record, `oSearchResolver` returns the address unchanged and the next resolver in the chain gets a turn — for example, an address that already carries its own transports needs no lookup. The leader registry is the authoritative source for capability and address discovery; libp2p's Kademlia DHT handles low-level peer routing underneath, not capability lookup.
  </Step>
  <Step title="Walk the hierarchy">
    For nested addresses (`o://parent/child`), routing walks parent pointers. A request to `o://parent/child` first resolves the parent, then asks the parent to route to its child.
  </Step>
</Steps>

This means a node only needs to know one address to find every other node: the leader's. The rest of the topology unfolds from there.

## How a node joins a network

The join sequence is short, and almost all of it is `oCore`'s lifecycle machinery doing its job.

<Frame>
```
construct        →   instantiate the class, store config
start()          →   run the lifecycle (see /concepts/lifecycle)
  libp2p up      →   listeners bind, peerId committed, multiaddrs populated
  register       →   POST registration to leader registry
  register       →   POST registration to parent (if any)
state = RUNNING  →   the node is reachable and discoverable
```
</Frame>

If the leader or parent is briefly unreachable, registration fails silently and the node still reaches `RUNNING`. See [the lifecycle page](/concepts/lifecycle) for why.

## Different node shapes

The class hierarchy stays the same; the libp2p config changes.

<CardGroup cols={2}>
  <Card title="oServerNode" icon="server">
    Listens on TCP, WebSocket, and IPv6. The default for any node that should accept dials from anywhere on the network. `oNodeTool` extends this.
  </Card>
  <Card title="oClientNode" icon="laptop">
    No listeners. Dial-only. Useful for short-lived scripts that connect, do work, and disconnect.
  </Card>
  <Card title="oWebsocketNode" icon="globe">
    Listens on WS only. Used when the node sits behind a proxy or load balancer that terminates TCP for you.
  </Card>
  <Card title="oLimitedTool" icon="mobile">
    Empty listeners, `runOnLimitedConnection: true`, circuit-relay-v2 enabled. The browser and NAT-restricted shape — reaches the network through a relay.
  </Card>
</CardGroup>

All four shapes are interchangeable from the caller's perspective. A request to `o://my-node` does not know or care which one is on the other end; the leader registry hands back whichever multiaddrs work.

## Leader vs worker

A common point of confusion: leaders and workers are **the same class of thing**. There is no `LeaderNode` superclass and no special privilege on the wire. A leader is just a node with two extra responsibilities:

- It is the **root of the hierarchy** — its address is configured on every other node as the bootstrap pointer.
- It **hosts the registry** — a child node at `o://leader/registry` that other nodes write to and query.

Workers register with leaders. Leaders accept registrations. Both run the same lifecycle, the same `use()` primitive, the same state machine. Demoting a leader to a worker is a config change, not a code change.

<Note>
  In production deployments, the leader is often itself a small cluster (multiple `oLeaderNode` instances behind a stable address) for HA. The contract above does not change.
</Note>
