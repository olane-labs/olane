# Emergent Workflows vs Prebuilt Workflows

## The Problem with Prebuilt Workflows

Most AI agent frameworks (LangGraph, n8n, CrewAI, OpenAI's Agent Builder, Microsoft Semantic Kernel) require you to explicitly define workflows upfront.

### LangGraph (StateGraph)

```typescript
// You must pre-define the entire workflow structure
const workflow = new StateGraph({
  nodes: ['fetch_data', 'analyze', 'report'],
  edges: [
    ['fetch_data', 'analyze'],
    ['analyze', 'report']
  ]
});
```

### n8n (Visual DAG)

You drag nodes onto a canvas and connect them with edges. The workflow is fixed at design time - if requirements change, you must manually rewire the canvas.

### CrewAI (Fixed Roles)

```python
# Pre-defined crew with fixed roles
crew = Crew(
  agents=[researcher, analyst, writer],
  tasks=[research_task, analysis_task, writing_task],
  process=Process.sequential  # Or hierarchical
)
```

### Problems with Prebuilt Workflows

| Problem | Impact |
|---------|--------|
| **Rigid** | Can't adapt to unexpected situations or changing requirements |
| **Brittle** | Edge cases require creating entirely new graphs/flows |
| **Hard to Reproduce** | Re-running a flow doesn't give you a portable, addressable record of the exact execution |
| **High Maintenance** | Every workflow change requires updating graph definitions |
| **Complex Coordination** | Multi-agent workflows become tangled spaghetti |

## Olane's Emergent Workflow Solution

Instead of pre-defining workflows, Olane enables **emergent orchestration** where the execution path is discovered at runtime through an evaluate/execute loop.

### How It Works

```typescript
// Send high-level intent - no predefined workflow needed
const result = await node.use(new oAddress('o://finance/analyst'), {
  method: 'intent',
  params: {
    intent: 'Analyze Q4 2024 revenue and generate executive PDF report'
  }
});
```

### What Happens Behind the Scenes

The tool node autonomously discovers an execution path. The default lane
loop alternates between two capabilities — `EVALUATE` (reason about the
current state and intent) and `EXECUTE` (run a tool) — re-evaluating after
each step until it decides to stop or hits `MAX_CYCLES` (default 20):

```
Cycle 1: EVALUATE → "I need revenue data for Q4 2024; o://data/financial has a revenue tool"
Cycle 2: EXECUTE  → Fetch Q4 revenue data
Cycle 3: EVALUATE → "Data retrieved, need to analyze trends"
Cycle 4: EXECUTE  → Calculate growth rates and trends
Cycle 5: EVALUATE → "Analysis complete, need to format as PDF"
Cycle 6: EXECUTE  → Generate executive summary PDF
Cycle 7: EVALUATE → STOP: report complete, returning result to agent
```

### Key Benefits

| Benefit | Description |
|---------|-------------|
| **Adaptive** | Responds to actual context and requirements in real-time |
| **Resilient** | Handles edge cases without code changes |
| **Replayable** | Each run is content-addressed (CID), persisted, and can be replayed deterministically by CID |
| **Maintainable** | Intent stays the same, execution evolves |
| **Simple** | No complex graph definitions to maintain |

## How Emergent Workflows Work

### 1. Capability Loop

Instead of pre-defined graphs, tool nodes use an **evaluate → execute → re-evaluate → stop** cycle:

```typescript
class FinancialNode extends oLaneTool {
  // No workflow graph needed - just define capabilities
  
  async _tool_fetch_revenue(request) { /* ... */ }
  async _tool_analyze_trends(request) { /* ... */ }
  async _tool_generate_report(request) { /* ... */ }
}

// Intent triggers autonomous capability loop
await node.use(new oAddress('o://finance/analyst'), {
  method: 'intent',
  params: { intent: 'Create Q4 financial report' }
});
```

The capability loop:
1. **EVALUATE**: Analyze current state and intent, then decide the next step
2. **EXECUTE**: Run the chosen capability
3. **RECORD**: Append the step to the run history (persisted, content-addressed by CID)
4. **RE-EVALUATE**: Done? If no, return to EVALUATE
5. **STOP**: Bounded by `MAX_CYCLES` (default 20)

### 2. Persisted, Replayable Runs

Each lane run is recorded step by step and stored as a **content-addressed
object (CID)**. Because the full sequence is persisted, any run can be
replayed deterministically by its CID — useful for auditing, debugging, and
reproducing an exact execution:

```typescript
// Run an intent; the resulting lane is persisted and addressed by a CID
const result = await node.use(new oAddress('o://finance/analyst'), {
  method: 'intent',
  params: { intent: 'Analyze Q4 sales' }
});
const cid = result.result.data.cid;

// Later, replay that exact run deterministically by CID
await node.use(new oAddress('o://finance/analyst'), {
  method: 'replay',
  params: { cid }
});
```

### 3. Auditable Execution History

Because runs are persisted and content-addressed, executions are inspectable
and reproducible across the system:

```
┌─────────────────────────────────────────────┐
│  Agent: "Analyze customer churn"            │
│  → Run persisted, addressed by CID          │
└─────────────────────────────────────────────┘
              ⬇ replay by CID
┌─────────────────────────────────────────────┐
│  Deterministic replay of the exact run       │
│  → Inspect each evaluate/execute step        │
│  → Reproduce results for audit/debugging     │
└─────────────────────────────────────────────┘
```

### 4. Dynamic Discovery

Tool nodes find and coordinate with other nodes as needed:

```typescript
// Coordinator node doesn't know about workers upfront
class CoordinatorNode extends oLaneTool {
  async _tool_intent(request) {
    // Dynamically discover available nodes via the leader-hosted registry
    const dataNodes = await this.use(new oAddress('o://leader/registry'), {
      method: 'search',
      params: { capability: 'data_collection' },
    });

    const analysisNodes = await this.use(new oAddress('o://leader/registry'), {
      method: 'search',
      params: { capability: 'analysis' },
    });

    // Coordinate discovered nodes — every call goes through node.use(address, ...)
    const data = await this.use(new oAddress(dataNodes.result.data[0].address), {
      method: 'collect',
    });
    const insights = await this.use(
      new oAddress(analysisNodes.result.data[0].address),
      { method: 'analyze', params: { data: data.result.data } },
    );
  }
}
```

## Real-World Example: Financial Reporting

### Scenario
Generate quarterly financial reports with varying requirements based on quarter, regions, and forecast needs.

### Prebuilt Approach (LangGraph/n8n)

You must explicitly define every possible path:

```typescript
// Must handle all edge cases upfront
if (quarter === 'Q4') {
  workflow = annualReportWorkflow;
} else {
  workflow = standardReportWorkflow;
}

if (needsForecast) {
  addNode('forecast_node');
  addEdge('analyze', 'forecast');
}

if (multipleRegions) {
  addNode('consolidate_regions');
  addEdge('fetch', 'consolidate');
}

if (complianceRequired) {
  addNode('compliance_check');
  addEdge('report', 'compliance');
}

// Now execute the pre-defined workflow
await workflow.execute();
```

**Problems:**
- New requirement = new graph modifications
- Edge case handling becomes complex
- No reuse across similar workflows
- Manual maintenance burden grows

### Emergent Approach (Olane)

Same intent works for all scenarios:

```typescript
// Works for any quarter, any regions, any requirements
await node.use(new oAddress('o://finance/analyst'), {
  method: 'intent',
  params: {
    intent: 'Generate comprehensive Q4 financial report with forecasts for all regions'
  }
});
```

**What happens autonomously** (the loop alternates `EVALUATE` and `EXECUTE`):

```
Cycle 1:  EVALUATE → "This is Q4, need annual report format; fetch all-region data"
Cycle 2:  EXECUTE  → Fetch multi-region financial data
Cycle 3:  EVALUATE → "Multiple regions detected, need consolidation"
Cycle 4:  EXECUTE  → Consolidate regional data
Cycle 5:  EVALUATE → "Intent mentions forecasts; generate forecasts"
Cycle 6:  EXECUTE  → Generate forecasts
Cycle 7:  EVALUATE → "Annual report may need compliance checks"
Cycle 8:  EXECUTE  → Run compliance checks
Cycle 9:  EVALUATE → "Ready to assemble the report"
Cycle 10: EXECUTE  → Generate final report
Cycle 11: EVALUATE → STOP: complete
```

**Benefits:**
- ✅ Adapts to Q4 annual requirements automatically
- ✅ Detects multi-region scenario without being told
- ✅ Discovers forecasting requirement from intent
- ✅ Adds compliance checks based on context
- ✅ The run is persisted and can be replayed deterministically by CID

## When to Use Emergent vs Prebuilt

### Use Prebuilt Workflows When:

- ✅ **Workflow is truly fixed** (regulatory/compliance requirements)
- ✅ **No adaptation needed** (same steps every time)
- ✅ **Simple sequential flow** (A → B → C, never changes)
- ✅ **Path must be fixed up front** (the exact step sequence is mandated, not discovered)

### Use Emergent Workflows When:

- ✅ **Requirements vary** (different contexts need different approaches)
- ✅ **Edge cases common** (many special cases to handle)
- ✅ **Reproducibility matters** (runs should be persisted and replayable for audit/debug)
- ✅ **Dynamic coordination** (don't know all tools upfront)
- ✅ **Agent-agnostic** (serve both human and AI agents)

## Implementation Guide

### Step 1: Define Tool Capabilities

Instead of defining workflows, define atomic capabilities:

```typescript
// financial.methods.ts
export const FINANCIAL_METHODS = {
  fetch_revenue: {
    name: 'fetch_revenue',
    description: 'Fetch revenue data for date range',
    parameters: [/* ... */]
  },
  analyze_trends: {
    name: 'analyze_trends',
    description: 'Analyze growth trends in financial data',
    parameters: [/* ... */]
  },
  generate_report: {
    name: 'generate_report',
    description: 'Generate formatted financial report',
    parameters: [/* ... */]
  }
};
```

### Step 2: Build Tool Node

```typescript
class FinancialNode extends oLaneTool {
  constructor() {
    super({
      address: new oAddress('o://finance/analyst'),
      methods: FINANCIAL_METHODS,
      laneContext: {
        domain: 'Financial Analysis',
        expertise: ['Revenue', 'Trends', 'Reporting']
      }
    });
  }

  async _tool_fetch_revenue(request) { /* ... */ }
  async _tool_analyze_trends(request) { /* ... */ }
  async _tool_generate_report(request) { /* ... */ }
}
```

### Step 3: Send Intent

```typescript
// Agent sends intent - workflow emerges
const result = await node.use(new oAddress('o://finance/analyst'), {
  method: 'intent',
  params: {
    intent: 'Analyze Q4 revenue and create PDF report',
    context: 'Focus on YoY growth trends'
  }
});
```

### Step 4: Persist and Replay

No additional code needed - the system automatically:
- Records each evaluate/execute step of the run
- Stores the full run as a content-addressed object (CID)
- Lets you replay any run deterministically by its CID
- Keeps runs inspectable for auditing and debugging

## Best Practices

### 1. Design Intent-First

Think about what agents want to accomplish, not the steps:

```typescript
const target = new oAddress('o://data/pipeline');

// ❌ Step-focused (prebuilt mindset)
await node.use(target, { method: 'step1_fetch' });
await node.use(target, { method: 'step2_transform' });
await node.use(target, { method: 'step3_report' });

// ✅ Intent-focused (emergent mindset)
await node.use(target, {
  method: 'intent',
  params: { intent: 'Transform data and create report' }
});
```

### 2. Build Atomic Capabilities

Each tool should do one thing well:

```typescript
// ✅ Atomic capabilities
async _tool_fetch_data(request) { /* ... */ }
async _tool_validate_data(request) { /* ... */ }
async _tool_transform_data(request) { /* ... */ }
async _tool_save_data(request) { /* ... */ }

// System composes them as needed
```

### 3. Provide Rich Context

Help the system make good decisions:

```typescript
const node = new AnalyticsNode({
  laneContext: {
    domain: 'Customer Analytics',
    expertise: ['Churn Analysis', 'LTV', 'Segmentation'],
    constraints: ['GDPR compliant', 'Real-time only'],
    preferences: ['Prioritize accuracy over speed']
  }
});
```

### 4. Trust the Loop

Don't micromanage - let the evaluate/execute loop find the path:

```typescript
const target = new oAddress('o://analytics/customer');

// ❌ Over-specified intent (defeats the purpose)
await node.use(target, {
  method: 'intent',
  params: {
    intent: 'First fetch data, then filter by date, then calculate averages, then...'
  }
});

// ✅ High-level intent (enables emergence)
await node.use(target, {
  method: 'intent',
  params: {
    intent: 'Analyze customer retention trends for Q4'
  }
});
```

## Summary

**Prebuilt workflows** (LangGraph, n8n, CrewAI) require you to explicitly define every workflow upfront, making them rigid and brittle.

**Emergent workflows** (Olane OS) discover a path through an evaluate/execute loop, adapt to changing requirements automatically, and persist each run as a content-addressed object that can be replayed deterministically by CID.

Choose emergent workflows when you want adaptive, resilient systems with reproducible runs. Choose prebuilt workflows when you need deterministic, regulatory-compliant fixed processes.
