---
title: "Nodes, tools, and tool nodes"
description: "How the Node, Tool, LaneTool, and McpTool hierarchy maps to your code."
---

A **node** is a network identity. A **tool** is a node with declared methods. A **lane tool** is a tool that resolves natural-language intent. An **MCP tool** is a tool that bridges an external MCP server. They form a single inheritance ladder, and you pick the rung that matches what you are building.

## The hierarchy

The ontology is enforced in code. Each class is a strict superset of the one above it.

<Frame>
```
Thing
 └── Node              has Address, can send/receive over libp2p
      └── Tool         has Methods (_tool_*), can be invoked
           ├── LaneTool   has intent resolution (capability loop)
           └── McpTool    bridges an external MCP server as methods
```
</Frame>

In the source tree:

- **`oCore`** (`@olane/o-core`, `packages/o-core/src/core/o-core.ts`) — abstract base. Owns lifecycle, `use()`, the state machine.
- **`oToolBase`** (`@olane/o-tool`, `packages/o-tool/src/o-tool.base.ts`) — adds method discovery via the `_tool_` prefix.
- **`oTool(Base)`** — mixin that adds built-in methods (`ping`, `stop`, `route`, `hello_world`, `index_network`).
- **`oNode`** (`@olane/o-node`, `packages/o-node/src/o-node.ts`) — adds libp2p.
- **`oServerNode`** (`packages/o-node/src/nodes/server.node.ts`) — listens on TCP, WS, and IPv6.
- **`oNodeTool`** (`packages/o-node/src/o-node.tool.ts`) — `oTool(oServerNode)`. The default base for most nodes.
- **`oLaneTool`** (`@olane/o-lane`, `packages/o-lane/src/o-lane.tool.ts`) — `withLane(oNodeTool)`. Adds `_tool_handshake`, `_tool_intent`, and the capability loop.

## Choosing a base class

Pick by what callers send to your node, not by how many methods it has.

| If your node has… | Pick |
| --- | --- |
| 1–5 well-defined methods, callers know the method names | `oNodeTool` from `@olane/o-node` |
| 5+ methods, callers send natural-language intent | `oLaneTool` from `@olane/o-lane` |
| A bridge to an external MCP server | `McpBridgeTool` from `@olane/o-mcp` |
| A browser, mobile, or NAT-restricted client | `oLimitedTool` from `@olane/o-client-limited` |

The number of methods is a hint, not a rule. The real question is whether callers know which method to call. If yes, `oNodeTool`. If they want to say "analyze Q4 financials" and let the node decide, `oLaneTool`.

## What every tool node gets for free

The `oTool` mixin injects five methods into every tool node. You do not write them; you call them.

<ParamField path="_tool_ping" type="method">
  Liveness check. Returns `{ pong: true, timestamp }`. Used by the leader for health.
</ParamField>
<ParamField path="_tool_stop" type="method">
  Remote shutdown. Triggers the node's `stop()` sequence.
</ParamField>
<ParamField path="_tool_route" type="method">
  Forwards a request to a child node by relative address. Used by parent-child routing.
</ParamField>
<ParamField path="_tool_hello_world" type="method">
  Trivial connectivity test. Returns `{ message: 'Hello, world!' }`.
</ParamField>
<ParamField path="_tool_index_network" type="method">
  Triggers re-indexing of this node's tools into the leader's registry.
</ParamField>

These five are the reason every node is at least slightly callable from day one — even if you have not declared a single method of your own, agents can ping you, route through you, and ask the network to index you.

## What `oLaneTool` adds

`oLaneTool` (from `withLane(oNodeTool)`) adds two more methods and a runtime loop.

<ParamField path="_tool_handshake" type="method">
  Returns the node's lane context: domain, expertise, available methods. Agents call this to learn what a node can do before sending an intent.
</ParamField>
<ParamField path="_tool_intent" type="method">
  The entry point for natural-language goals. The node receives `{ intent: string, context?: object }` and runs the **capability loop**: evaluate, choose a tool, execute, evaluate, repeat, stop. Internally, the loop uses an LLM to plan over the methods declared by the node.
</ParamField>

In other words: every `oNodeTool` is callable by name. Every `oLaneTool` is also callable by description.

## A minimal node, three ways

The same domain expressed at three rungs of the ladder.

<CodeGroup>
```typescript oNodeTool — direct call
import { oNodeTool } from '@olane/o-node';
import { oAddress } from '@olane/o-core';
import { oRequest } from '@olane/o-protocol';

class ConverterNode extends oNodeTool {
  constructor() {
    super({ address: new oAddress('o://utilities/converter') });
  }

  async _tool_convert(request: oRequest) {
    const { amount, from, to } = request.params;
    return { result: amount * 1.18, from, to };
  }
}

// Caller knows the method:
await node.use(new oAddress('o://utilities/converter'), {
  method: 'convert',
  params: { amount: 100, from: 'EUR', to: 'USD' },
});
```

```typescript oLaneTool — intent
import { oLaneTool } from '@olane/o-lane';
import { oAddress } from '@olane/o-core';

class AnalystNode extends oLaneTool {
  constructor() {
    super({
      address: new oAddress('o://company/finance/analyst'),
      laneContext: { domain: 'Financial Analysis' },
    });
  }

  async _tool_calculate_revenue(request) { /* ... */ }
  async _tool_calculate_expenses(request) { /* ... */ }
  async _tool_generate_report(request) { /* ... */ }
}

// Caller describes the goal:
await node.use(new oAddress('o://company/finance/analyst'), {
  method: 'intent',
  params: { intent: 'Analyze Q4 2024 and summarize.' },
});
```

```typescript McpBridgeTool — external MCP
import { McpBridgeTool } from '@olane/o-mcp';
import { oAddress } from '@olane/o-core';

class GithubBridge extends McpBridgeTool {
  constructor() {
    super({
      address: new oAddress('o://bridges/github'),
      mcpServer: {
        command: 'npx',
        args: ['-y', '@modelcontextprotocol/server-github'],
      },
    });
  }
}

// The bridge exposes every MCP tool as an Olane method.
```
</CodeGroup>

## Mental model

Think of the ladder like inheritance in any object system.

<CardGroup cols={2}>
  <Card title="Node = identity" icon="at">
    Has an address, a peerId, multiaddrs. Can send and receive. Cannot do anything domain-specific.
  </Card>
  <Card title="Tool = identity + verbs" icon="wrench">
    A node that declares methods. Now callable by name.
  </Card>
  <Card title="LaneTool = tool + planner" icon="brain">
    A tool with a capability loop. Now callable by description.
  </Card>
  <Card title="McpTool = tool + adapter" icon="plug">
    A tool whose methods are forwarded to an external MCP server.
  </Card>
</CardGroup>
